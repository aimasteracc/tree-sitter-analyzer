# Tree-Sitter-Analyzer：AI 代码上下文引擎战略

状态：草案
受众：产品、工程、内部推动者
最后更新：2026-03-26

## 执行摘要

Tree-Sitter-Analyzer 不应该被定位为"一个基于 tree-sitter 的代码分析工具"。

这种定位低估了这个仓库中已经存在的东西。该产品已经结合了：

- 使用 `fd` 和 `ripgrep` 的快速仓库检索
- 跨 17 种语言的基于 AST 的结构化分析
- 通过 MCP 集成到 Claude Desktop、Cursor 和 Roo Code
- 项目边界强制和本地优先的安全控制
- 针对大文件和大型仓库的令牌优化工作流

更准确的类别定义是：

`Tree-Sitter-Analyzer 是一个面向 AI 辅助开发的本地优先代码上下文引擎。`

它的工作不只是解析代码。它的工作是帮助人类和 AI 代理安全、快速、结构化地精准获取他们真正需要的代码上下文。

## 我们应该占领的类别

### 弱定位

- 代码分析工具
- Tree-sitter 实用工具
- 更好的代码 grep
- 大文件的 MCP 助手

### 强定位

- 本地优先代码上下文引擎
- AI 代理的安全上下文基础设施
- 大型代码库的检索 + 结构层
- 企业仓库的 AI 就绪代码理解管道

## 为什么这很重要

现有的故事是准确的，但格局太小。它解释了产品是如何诞生的，但没有说明机会的规模。

真正的机会更大：

1. AI 编码工具仍然需要搜索。
2. 大型仓库仍然会破坏天真的"将整个文件粘贴到模型中"的工作流。
3. 安全敏感团队仍然无法将源代码发送到远程索引服务。
4. 单纯的快速文本检索是不够的；AI 还需要结构化切片和令牌控制。

Tree-Sitter-Analyzer 已经处于这个交叉点。

## 产品论点

### 一句话论点

Tree-Sitter-Analyzer 是一个本地优先的代码上下文引擎，它结合了快速检索、基于 AST 的结构和安全交付，使 AI 工具能够有效地处理大型代码库。

### 扩展论点

在现代 AI 辅助开发中，瓶颈不再只是解析代码。瓶颈是以足够快、足够安全的方式，以模型实际可用的形式，将正确的上下文提供给模型。

Tree-Sitter-Analyzer 通过将仓库探索转变为管道来解决这个问题：

`找到正确的文件 -> 找到正确的匹配 -> 提取正确的结构 -> 只发送正确的上下文`

这是一个产品类别，而不是一个实用功能。

## 已经存在的东西

当前的仓库已经支持这个故事的核心。

### 1. 检索层

- `list_files` 用于边界范围内的文件发现
- `search_content` 用于内容搜索
- `find_and_grep` 用于两阶段搜索工作流

### 2. 结构理解层

- 通过 tree-sitter 进行基于 AST 的分析
- 查询驱动的方法、类、函数、导入和其他代码元素的提取
- 针对大文件工作流的部分读取和结构表输出

### 3. 交付层

- 与 Claude Desktop、Cursor 和 Roo Code 的 MCP 服务器集成
- CLI 和 Python API 作为共享核心能力之上的并行接口

### 4. 安全层

- 项目边界强制
- 路径遍历防护
- 输入清理
- 错误清理
- 本地优先操作模型

### 5. 令牌控制层

- `count_only`
- `summary_only`
- `group_by_file`
- `total_only`
- `suppress_output` 加 `output_file`

这已经更接近于一个上下文引擎，而不是一个解析器包装器。

## 战略重新定位

### 当前隐含的信息

"我们构建了一个有用的内部工具，因为其他搜索工具太慢、不安全或不完整。"

### 更好的信息

"我们构建了一个本地优先的代码上下文引擎，因为 AI 辅助开发在大型、敏感的仓库上会失败，除非检索、结构和安全被一起设计。"

这种定位更有雄心、更有防御性、更经得起未来考验。

## 切入点、扩展和最终目标

### 初始切入点

首先在最痛苦的用例中获胜：

- 大型 Java 仓库
- 遗留或迁移系统，如 COBOL 到 Java 的转换
- 事故响应和影响分析
- 需要针对性上下文而不是原始文件转储的 AI 编码助手

### 扩展路径

在证明切入点之后，将同样的引擎扩展到：

- 一般的大型仓库 AI 开发
- 跨语言代码导航
- 企业本地 AI 工作流
- 内部工程团队的安全 MCP 基础设施

### 最终目标

成为位于代码库和 AI 代理之间的标准本地上下文层。

长期竞争不是"其他 tree-sitter 工具"。而是任何试图控制 AI 如何获取代码上下文的系统。

## 从快速正则搜索中学到的经验

Cursor 的快速正则搜索方向的重要教训不是"不惜一切代价让 `rg` 更快"。

教训是：

`搜索不是 AI 代理的附件；它是核心基础设施。`

Tree-Sitter-Analyzer 应该吸收这个教训，同时不失去其差异化。

### 应该复制的

- 将检索延迟视为一流的产品关注点
- 在非常大的仓库上精确正则扫描之前添加候选生成层
- 保持整个工作流本地优先
- 为完整的 AI 工作流优化，而不仅仅是孤立的基准胜利

### 不应该复制的

- 不要将产品压缩为"更快的正则搜索"
- 不要为了追求全局索引而削弱项目边界控制
- 不要让词法检索掩盖基于 AST 的结构精度

## 产品架构方向

当前的检索管道可以演进而不是替换。

### 当前形态

`fd/list_files -> rg/search_content/find_and_grep -> tree-sitter query/partial read -> AI`

### 下一代形态

`边界范围的文件集 -> 本地文本索引候选过滤器 -> rg 精确匹配 -> tree-sitter 结构提取 -> 令牌优化的 AI 上下文`

### 要引入的新层

推荐的服务名称：

- `CandidateSearchService`

可接受的替代方案：

- `TextIndexService`

### 新层的角色

- 仅在项目边界内操作
- 为大型仓库搜索构建本地候选集
- 喂养精确匹配工具而不是替换它们
- 在 AST 提取之前减少延迟
- 保留现有的 CLI 和 MCP 行为作为默认路径

## 三个具体的产品赌注

### 赌注 1：重新定位产品

将主要叙述从"tree-sitter 分析"改为"AI 代码上下文引擎"。

这是最高杠杆的举措，因为它改变了对每个功能的理解方式。

### 赌注 2：添加可选的候选生成层

在 `search_content` 和 `find_and_grep` 之前为大型仓库引入本地索引或预过滤路径。

这应该是增量的、实验性的，最初是可选的。

### 赌注 3：将 SMART 转变为产品原则

SMART 工作流不仅仅是文档。它是操作模型：

`Set -> Map -> Analyze -> Retrieve -> Trace`

这是教导用户、AI 代理和内部团队如何使用平台的最自然方式。

## 信息支柱

### 支柱 1：快速查找

在大型仓库中快速查找相关文件和匹配，而不是盲目扫描整个世界。

### 支柱 2：精确读取

使用 AST 感知结构和针对性提取，而不是将完整文件转储到模型中。

### 支柱 3：本地安全

将代码和分析保持在本地，在明确的项目边界内，具有企业友好的安全控制。

## 推荐的一句话描述

Tree-Sitter-Analyzer 是一个面向 AI 辅助开发的本地优先代码上下文引擎，结合了快速仓库检索、基于 AST 的结构化分析和安全的 MCP 集成。

## 推荐的介绍文案

Tree-Sitter-Analyzer 是一个面向大型和复杂代码库的本地优先代码上下文引擎。它通过结合快速搜索、基于 AST 的结构化分析和严格的项目边界控制，帮助开发者和 AI 编码工具只检索他们真正需要的代码上下文。

它源于一个真实的约束：由 COBOL 到 Java 转换产生的超大型 Java 文件无法有效地交给 AI 模型，发送整个文件既浪费令牌又产生上下文噪音。Tree-Sitter-Analyzer 通过将源代码分解为有意义的结构单元并实现精确检索而不是原始文件转储来解决这个问题。

除了解析之外，Tree-Sitter-Analyzer 将文件发现、内容搜索、结构查询和部分提取统一到一个本地优先的工作流中。这提高了调查速度，降低了 AI 令牌成本，并提高了事故响应、影响分析、遗留现代化和 AI 辅助开发的代码理解质量。

所有处理都在本地运行。源代码和分析结果不需要离开环境，搜索范围在项目边界内保持可控。这使得该产品对企业和内部工程环境特别有力。

## 我们可以捍卫的证明点

- 跨企业、系统、Web、数据和配置域的 17 种语言支持
- MCP 集成被设计为一流接口，而不是事后考虑
- 两阶段搜索已经存在于产品界面中
- 令牌优化是一个有文档记录的产品功能
- 安全边界在设计和测试中都是明确的
- CLI、MCP 和 Python API 共享一个通用的平台核心
