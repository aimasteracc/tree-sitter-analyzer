# 代码理解报告 - 检查清单（通用模板）

> 生成「xxx_理解报告.md」时按此清单核对，避免遗漏。  
> 报告只输出到**用户指定路径所在文件夹**，不写入 V2 仓库。  
> 本文件为通用模板，不包含任何具体项目/文件的业务内容。

---

## 一、必选结构

- [ ] **5 分钟概览**：功能一句话、规模（函数/类/行）、主要依赖
- [ ] **15 分钟架构**：函数/类清单与职责、调用关系、数据流、Mermaid 图
- [ ] **30 分钟深入**：业务规则、输出物、关键设计点、入口/命令行
- [ ] **学习/修改建议顺序**

---

## 二、若代码中存在，建议单独小节说明

- [ ] **继承/关联机制**：若有「由配置或文档声明 A 继承 B、自动复制某类字段」等逻辑，写清触发条件、继承内容、实现位置
- [ ] **外部数据补全**：若有从 Excel/DB/配置文件等「反向补全到内存结构」的逻辑，写清数据来源、目的、与主数据源的优先级
- [ ] **导航/链接**：若有主视图↔子视图、文档↔代码等双向跳转或超链接，写清两端如何互链、失败时的降级方式
- [ ] **某类用例的特殊分支**：若有按类型（如性能测试、异常测试）走不同解析或展示的逻辑，写清判定条件与差异
- [ ] **序号/顺序管理**：若有文档内顺序、过滤用序号等，写清含义、存储位置、用途
- [ ] **日志/输出分层**：若有「仅文件 / 终端+文件」等分层，写清各自用途、路径规则、平台相关处理（如编码）

---

## 三、容错与实现细节（若有则写）

- [ ] **路径/配置解析**：是否有多层目录搜索、多候选路径、缺省路径
- [ ] **命名与长度限制**：是否有长度截断、保留字/重名时的后缀或重命名规则
- [ ] **缺文件/缺配置**：缺失时的降级策略与日志

---

## 四、可选（视需要）

- [ ] 错误与异常：关键文件不存在、格式错误、重复 ID 等的处理
- [ ] 性能与规模：大数据量时的注意点、内存或文件大小
- [ ] 维护与扩展：如何加新配置项、新格式、新输出样式

---

## 五、报告输出约定

- 文件名：`{分析对象名}_理解报告.md`
- 位置：**用户指定的文件/项目所在目录**，不写入 V2
- 机密项目：全文仅落在用户自己的路径下

---

使用方式：下次说「用 V2 工具做理解报告，按检查清单输出」时，按此**通用**清单逐项检查，具体内容根据**当前被分析的文件/项目**填写，不把其他项目的内容写进本清单。

---

## 六、用来理解「我们自己的 V2 项目」时

- **工具组合**：与理解外部项目相同 —— `analyze_code_structure`、`analyze_code_graph`、`visualize_code_graph`、必要时 `read_file`。
- **模板**：同一份 MD 模板（5/15/30 分钟 + 特殊机制 + Mermaid + 学习顺序 + 工具记录）。
- **粒度**：
  - **单文件**：对 `mcp/server.py`、`features/instant_understanding.py`、`core/parser.py` 等关键文件逐个分析，可各生成一份「xxx_理解报告.md」。
  - **项目级**：汇总多文件 + 目录结构，生成一份「V2_理解报告.md」。
- **报告位置**：V2 自己的报告可放在 V2 仓库内（如 `docs/V2_理解报告.md`）；外部/机密项目的报告只放在**对方指定路径**，不写回 V2。
- **示例**：已生成 `docs/V2_理解报告.md`，可作为「用工具组合 + 模板理解 V2」的参考。
