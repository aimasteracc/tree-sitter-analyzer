## Summary

We will **not** ship a general-purpose code-visualization product. We will ship **one** thing: a **correctness-credibility demo** — a single page (rendered HTML, generated by `viz action=demo`) that puts TSA's cross-language call graph side-by-side with at least one LLM-inferred competitor's graph on a **fixed sample corpus**, with every TSA edge tagged by provenance (file:line + AST node + edge-kind) so the difference is **visible at a glance**. That page becomes the README hero image. Mermaid/SVG/full interactive HTML are explicitly **deferred** to a future RFC, gated on whether this demo moves a measurable signal.

This is a deliberate rejection of the natural shape (3-emitter `tsa viz` → HTML/SVG/Mermaid stack that looks like dependency-cruiser). That shape ships in 3-5 engineer-weeks, lands at the dep-cruiser / madge star ceiling (~6-10k), dilutes TSA's "agent-native correctness" moat, and duplicates work UA (Understand-Anything) and CodeSee already did. The correctness-credibility demo ships in 1-2 weeks, exploits the only moat we actually have, and either earns its own follow-on RFC by moving the needle or stops cleanly.

## Motivation

### The investor hypothesis (explicit)

> "我看所有的可視化的項目都star多" — D3 (110k★), Excalidraw (123k★), Mermaid (88k★) all visualize.

### Why the surface form of this hypothesis is **vanity** (panel Critic, accepted)

D3 / Excalidraw / Mermaid are **domain-agnostic primitives** other people use to build their own visualizations — they are not "code visualization tools." The actual *code-viz* category caps far lower: Sourcetrail 16k (archived 2021), CodeCharta 458, dependency-cruiser 6.7k, code2flow 4.5k, pyan 700, CodeSee (closed, fading), Sourcegraph 10.3k (also closed-source 2024). The ceiling is real and 10-50× below the primitives. Optimizing for "ship viz → get D3-class stars" is optimizing for the wrong end of the causal chain.

The real chain is: **hero artifact → README screenshot → social/HN share → stars from people who never run the tool.** Stars come from a screenshot, not from feature count. The PM brief is therefore: *what is the one screenshot that, posted on HN / Twitter / a README header, would make a code-intelligence-curious developer star this repo?*

### Why the answer is **not** "self-contained interactive HTML"

R2 proposed a self-contained Cytoscape/D3 HTML report (the dep-cruiser shape) and called this Bet A ("AST-verified vs LLM-inferred"). Critic correctly flagged that this is structurally **Bet B**: the *artifact shape* is the same as dep-cruiser's; the differentiation lives only in the marketing language. A viewer looking at our HTML viz next to dep-cruiser's HTML viz cannot eyeball-distinguish "correct edges" from "plausible edges" — both look the same. **Correctness is invisible until proven by side-by-side diff.** A pretty-but-generic HTML report ships for 4-8 engineer-weeks, has zero unique visual element, and is the dep-cruiser ceiling on day one.

### What is left, then

There is exactly one viz shape no incumbent ships that TSA can ship and that *makes correctness visible*:

> **A side-by-side correctness diff between TSA's verified cross-language call graph and a competitor's LLM-inferred graph on a fixed reference corpus, where every TSA edge carries a click-through provenance pin (file:line + AST node + edge-kind), and a "diff" panel highlights edges TSA has that the competitor missed, edges the competitor has that TSA flags as spurious, and a count.**

The screenshot reads: "Understand-Anything shows you a graph. TSA shows you the RIGHT graph — here is the evidence, on a real repo, with every line citable." That is the hero. It is **defensible** (the data backing it is TSA's actual SQLite KG with `edges_by_kind` + provenance — nothing in the comp set has it), **measurable** (edge counts, false-positive counts, FFI-boundary captures), **falsifiable** (if the diff is small / unconvincing, the RFC stops), and **costs no new platform dependencies** (data already in SQLite; rendering is one HTML page).

It is also the **only** Bet A shape Critic could not refute.

### Architectural-strategy alignment

`rfcs/ROADMAP-beyond-codegraph.md` lists P0–P4 (token-cost win, cross-lang correctness moat, file-level stdlib, reactive push, resolution confidence). **Viz is not in P0–P4.** The investor's viz pitch is a new direction, and the only way it survives strategy review is by *amplifying* the correctness moat (P1) rather than competing for engineering weeks against it. The demo shape does exactly that — it ships the moat *as the hero*.

## Detailed design

### The deliverable, in one sentence

A single rendered HTML page, generated by `viz action=demo` (MCP) and `--viz-demo` (CLI), comparing TSA's cross-language call graph to one competitor's on the **same fixed corpus**, with edges color-coded by provenance and a diff panel quantifying disagreement.

### Architecture — what's new vs. what's reused

| Layer | New? | Notes |
|---|---|---|
| Call-graph data (nodes, edges, kinds, provenance) | **Reused** | Already in the SQLite KG. `edges_by_kind` exists. Provenance (file:line + AST node) is in the cache; today's viz emitters don't surface it. No DB change. |
| Rendering | **New, small** | One Jinja-style HTML template + one minimal JS embed (Cytoscape.js inlined, ~600KB — see Open Q #6). Single-file output, opens in any browser, no CDN. |
| Competitor data | **New, one-off** | Bundled JSON snapshot of one competitor's call graph on the same corpus, captured once at spec time and version-pinned. Refresh on demand, not in CI. |
| Diff engine | **New, small** | Pure-Python set diff over (caller, callee, edge_kind) tuples. ~150 LOC. |
| MCP surface | **New action on existing facade** | `viz action=demo` — read-only, returns the HTML as a content block (`text/html`) plus a JSON summary (counts, FFI captures, provenance density). No new facade, no new tool registration. |
| CLI surface | **New flag on existing group** | `--viz-demo [--corpus <name>] [--competitor <name>] [--out <path>]` — same shape as existing `--viz-*` flags. |

### Data store decision (investor constraint)

> Investor constraint (2026-06-19): "数据库可以使用我 wiki 中最具摄入的有一個開源的數據庫" → resolved to **LadybugDB** (the open-source graph DB used by GitNexus; the most-deeply-ingested DB material in the user's wiki, ~50+ source files under `wiki/raw/ai-tech/git-nexus/`).

**Decision: not triggered by this RFC.** The viz demo reads the existing SQLite knowledge graph directly via the in-process query layer; it persists no viz state. No new datastore is required, which keeps faith with the older LOCKED decision *"不换 TSA 主存储引擎"* (memory: `project_helixdb-assessment`, 2026-06-12). LadybugDB remains the **pre-approved escape hatch** for any future viz capability that genuinely needs graph-query semantics or persistent viz state (e.g. saved layouts, multi-user annotations, RFC-0001-driven live state) — pre-decided so a future RFC author does not re-pick. **This RFC does not invoke that escape hatch.** Any future RFC that proposes a new datastore must justify why SQLite is insufficient AND must use LadybugDB if SQLite truly is insufficient.

### MCP surface (facade + action)

Add **one** action to the existing `viz` facade (`viz_facade.py`). Do not add a new facade. Do not touch the other 3 actions.

```python
# viz_facade.py — augmented action map
_VIZ_DESCRIPTION = (
    "... existing 3 actions ...\n"
    "- action=demo — generate a correctness-credibility comparison page "
    "(TSA's cross-language call graph vs a competitor's LLM-inferred graph "
    "on a fixed sample corpus, with provenance-pinned edges). "
    "Returns the rendered HTML and a JSON summary of edge counts, FFI captures, "
    "and disagreement metrics. Params: corpus (str, default 'fixed_polyglot_v1'), "
    "competitor (str, enum: 'ua_snapshot_v1'), include_html (bool, default true)."
)
```

Return shape (TOON-wrapped per LOCKED §1 — MCP default `toon`):

```yaml
success: true
verdict: INFO
corpus: fixed_polyglot_v1
competitor: ua_snapshot_v1
summary:
  tsa_edges_total: 1234
  competitor_edges_total: 987
  edges_tsa_has_competitor_missing: 412
  edges_competitor_has_tsa_flags_spurious: 88
  ffi_boundaries_captured: 17       # cross-language calls (Python→Rust, Rust→C, …)
  provenance_density: 1.0           # 1.0 = every TSA edge has file:line+ast_node+kind
agent_summary:
  summary_line: "viz demo: 412 edges TSA has that UA misses, 17 FFI captures on fixed corpus"
  next_step: "Open the HTML file at <path> to see the side-by-side diff."
html_path: /tmp/tsa-viz-demo.html
html_size_bytes: 614400
```

The HTML itself is rendered to a file (not embedded in the response) — keeps TOON cheap, avoids "huge MCP payload" anti-pattern. `include_html=false` skips the file write for headless smoke tests.

### CLI surface (parity)

```bash
tsa --viz-demo                                                        # default corpus + competitor
tsa --viz-demo --corpus fixed_polyglot_v1 --competitor ua_snapshot_v1 # explicit
tsa --viz-demo --out docs/hero.html                                   # custom output path
```

Parity holds: same 3 user-facing args, same data path, same JSON summary on `--output-format json`. The MCP `action=demo` is the same code path with a different presentation layer (TOON wrapper).

### The HTML page (deliverable spec)

Single self-contained file, openable in any browser, no network at runtime.

- **Top strip**: project name, corpus name, generation timestamp, TSA version, competitor version.
- **Left panel**: TSA's call graph (Cytoscape.js force-directed, inlined). Nodes labeled `module::function`. Edges color-coded by kind (calls / extends / implements / imports). **Every edge** has a `data-provenance` attribute (`file:line@ast_node`) surfaced as a tooltip on hover and a clickable popover with the source snippet (snippet is bundled, no network call).
- **Right panel**: competitor's graph on the same corpus, same layout engine, no provenance (because the competitor doesn't have it). **Visual asymmetry is the point.**
- **Diff strip (bottom)**: three small bars — "TSA-only edges: 412", "Competitor-only edges: 88", "FFI boundaries captured: 17 vs 0". Each bar is clickable, filters both panels to the disagreeing subset.
- **Hero region (above the fold)**: one Python→Rust→C call chain, large, with the three provenance pins visible. *This is the screenshot.*

The page is the README hero image. **Before any code is written**, the spec writer (lead) must produce a paper/Figma mockup of the above-the-fold hero and validate against the falsifiability gate (Acceptance Criteria #1 below).

### The fixed corpus (deliverable spec)

One bundled-or-pinned reference repository, version-locked, polyglot enough to exhibit cross-language edges:

- Python + Rust (via PyO3 or maturin) + C (via cffi or ctypes) — at least 3 callable FFI boundaries.
- ~50-500 functions total (small enough to render readably, large enough to be non-trivial).
- Stable URL (a specific git SHA of an existing OSS project, or a small dedicated fixture repo under `tests/fixtures/viz_corpus/`).
- Indexed by TSA at spec-acceptance time; the indexed snapshot is committed under `tests/fixtures/viz_corpus_indexed/` (read-only).

Candidate corpora to evaluate during Phase 0 spike (Open Q #1):
- A small slice of the cryptography/pyca repo (Python + Rust + C via cffi).
- A small slice of the polars repo (Python + Rust).
- A bespoke 200-LOC fixture (fastest path, lowest realism).

Lead picks one at Phase 0 close. The choice is part of the spec, not implementation.

### The competitor snapshot (deliverable spec)

One competitor's call graph on the **same corpus**, captured once at spec time, committed under `tests/fixtures/viz_corpus_indexed/competitor_<name>_v1.json`. The format is a flat list of `{caller, callee, edge_kind?}` tuples — whatever the competitor exposes. Re-capture is a manual, periodic chore (not CI). The competitor identity is **deliberately not yet pinned** — Open Q #2 — but the leading candidate is Understand-Anything (15-54k★, May 2026 trending #1, explicitly LLM-inferred — the strongest contrast).

## Three-Surface impact (CLI ↔ MCP parity)

| Surface | New surface this RFC adds |
|---|---|
| CLI | `--viz-demo` (+ optional `--corpus`, `--competitor`, `--out`) |
| MCP | `viz action=demo` (+ same 3 optional params) |
| Library | `tree_sitter_analyzer.viz_demo.render(corpus: str, competitor: str, out: Path \| None) → DemoResult` (used by both surfaces; not a top-level public export) |

Both surfaces produce the **same** JSON summary (TOON-wrapped on MCP per LOCKED §1) and write the **same** HTML file. Parity test: a single integration test calls both surfaces against the same corpus and asserts the JSON summaries are byte-identical (modulo `output_format` wrapping).

## Drawbacks

- **Costs 6-9 engineer-days that could have gone into the documented P0-P4 roadmap.** This RFC frames the spend as "investor-directed strategic detour with a falsifiable kill-switch" — but it is a real opportunity cost against the correctness moat.
- **Hero-screenshot strategy is binary.** Either Phase 1 produces a screenshot that genuinely is striking (and we ride the social wave), or it produces a screenshot that looks like "another dep graph" (and we have shipped a 6-day artifact that earned no stars). The Acceptance Criteria gate (AC #1) reduces but does not eliminate this risk.
- **Comparison framing risks aggro from competitor.** A side-by-side that highlights UA's edge errors invites pushback. Mitigation: bundle the corpus snapshot version-pinned (not live), present diff as "snapshot from <date>", be transparent.
- **Maintenance debt of the bundled competitor snapshot.** Every TSA release that materially changes the call-graph extractor risks invalidating the demo. Mitigation: snapshot refresh is a manual, low-frequency step; the comparison is fundamentally about "TSA at vX vs competitor at vY on corpus at SHA Z" and that triple is fine to age.
- **Cytoscape.js bundle size (~600KB inlined).** Acceptable for a one-off HTML page; would not scale to "ship this in every viz response." This RFC's scope is **one** page, so this is fine here.

## Alternatives

- **Alternative A (R2's original): self-contained 3-emitter HTML viz product.** Pros: matches what the comp set ships. Cons: 3-5 engineer-weeks, dep-cruiser ceiling, dilutes the moat, R2's "AST-verified" framing collapses to marketing because the artifact looks identical to dep-cruiser. **Rejected** per Critic's Bet B analysis; revisitable if Phase 1 succeeds.
- **Alternative B: ship 3 emitters (Mermaid/SVG/HTML) per R3.** Pros: covers more surfaces. Cons: 3 serializers, golden-image flakiness in CI, lock-in to one frontend lib, ongoing maintenance of all three. **Rejected** — pick the right surface for the one hero, not three OK surfaces for nobody.
- **Alternative C: fix the 4 viz bugs (P0-A through P0-D), close the demo bet, return engineering to the P0-P4 moat.** Pros: no opportunity cost, leaves the existing viz surface healthier. Cons: zero star upside; the investor hypothesis goes unanswered; we never learn whether viz can move adoption. **Rejected** at brief stage — but it is the **fallback** if Phase 0 reveals Phase 1 is unaffordable, AND it is the **outcome** if Phase 1's falsifiability gate (AC #1) does not pass. So Alt C is the safety net, not the plan.
- **Alternative D: do nothing.** Pros: zero cost. Cons: investor signal unanswered; team learns nothing about whether viz mechanism applies to TSA. **Rejected** — investor explicitly directed PM to run this experiment.

## Prior art

- **GitNexus** — uses LadybugDB (the investor-pinned escape-hatch DB). Their MCP exposes `route_map`, `cypher` queries on a graph DB. They have not shipped a viz-credibility demo; their visualization is a generic web UI.
- **Understand-Anything (UA)** — current trending #1 (May 2026), 15-54k★ across forks. Pitch is explicit: "plain-English explanations of architecture." LLM-inferred graph; no provenance pins; no FFI-boundary capture surfaced. The **direct comparison target**.
- **CodeSee (closed)** — "impact maps" at the file level. Proprietary, LLM-augmented. Cannot inspect their edges; cannot diff against. Not a viable comparison target (no public snapshot).
- **codegraph (colbymchenry, MIT)** — the TS+SQLite+tree-sitter project documented in `wiki/codegraph-data-structures-report.md`. Provides a useful "reverse-comparison" target: same data approach as TSA, different language coverage. Possible secondary comparison if UA snapshot is unavailable.
- **dependency-cruiser (6.7k★)** — the canonical self-contained-HTML-report shape this RFC explicitly **declines** to copy.
- **Sourcetrail (16k★, archived 2021)** — proof that even the strongest native code-viz UX caps lower than D3/Mermaid/Excalidraw.

## Test plan (RED-first)

The failing tests we will write *first*. RED-first per the project's TDD discipline.

1. **`tests/regression/test_viz_correctness_demo.py::test_demo_smoke`** — calls `viz_demo.render(corpus='fixed_polyglot_v1', competitor='ua_snapshot_v1', out=None)`. 2. **`test_demo_ffi_capture_invariant`** — on the fixed corpus, `ffi_boundaries_captured` is exactly the documented number (per Phase 1.1 corpus pick). **Exact assertion** per LOCKED rule (no `>= N`).
3. **`test_demo_disagreement_nonzero`** — `edges_tsa_has_competitor_missing > 0` AND the disagreement set contains at least one FFI-boundary edge. (Without this, the hero point — "we capture FFI, they miss it" — is unproven.) **Exact lower bound is set at Phase 0 close**, per the "exact assertions only" rule (an exact pin against the measured corpus, not a hand-waved `>= 1`).
4. **`test_demo_cli_mcp_parity`** — both surfaces produce byte-identical JSON summaries on the same args.
5. **`test_demo_html_is_self_contained`** — the rendered HTML contains no `<script src=...>`, no `<link href=...>` to external URLs; opens with no network.
6. **`tests/unit/mcp/test_output_cost_invariants.py::test_demo_action_toon_le_json`** — extends the existing token-cost ratchet (Rule-11) to cover `viz action=demo`.

## What this RFC does NOT do (deferred)

Stated explicitly so reviewers do not expand scope mid-implementation:

- **Does not** add Mermaid emission for the demo (Mermaid is for RFC-0015's existing UML; the demo is interactive HTML by necessity).
- **Does not** introduce a general-purpose interactive HTML viz product.
- **Does not** integrate RFC-0001 Hyphae push (live diagram).
- **Does not** host any online demo (GitHub Pages or otherwise).
- **Does not** introduce LadybugDB or any new datastore (escape hatch documented above is for a *future* RFC).
- **Does not** add IDE-side rendering integration.
- **Does not** change existing `viz action=uml`, `viz action=graph`, `viz action=similarity` semantics (Phase 0 bug fixes are scope-separated PRs, each with their own RFC if non-trivial).
- **Does not** add a CI golden-image regression for the HTML (pixel-perfect HTML diff is famously flaky; we'll assert structural invariants instead — bundle size, presence of provenance attributes, edge counts in the embedded JSON).
- **Does not** ship a demo in any language other than the corpus's languages (Python + Rust + C in the candidate corpora).

## Open questions

The PM must resolve these *before* Phase 1 starts. Each maps to a Critic-flagged research gap.

1. **Corpus choice (Critic gap: "default-slice question").** cryptography/pyca? polars? bespoke 200-LOC fixture? Decision criterion: smallest realistic polyglot with ≥3 verifiable FFI boundaries; chooses-itself if any candidate fails to produce ≥10 cross-language edges through TSA's existing extractor. Lead decides at Phase 0 close.
2. **Competitor snapshot source (Critic gap: "UA correctness gap measured").** UA is the strongest contrast but needs an exportable graph. If UA does not expose its raw graph via API/MCP, fall back to codegraph (MIT, queryable, weaker LLM-inferred narrative). Lead spikes both during Phase 0 (1 day, time-boxed).
3. **Mermaid-in-claude.ai-chat empirical test (Critic gap).** Not strictly needed for this RFC (we emit HTML, not Mermaid), but the panel raised it as an unverified R3 claim that affects any future Mermaid-emitter RFC. Lead must run one tool call in claude.ai chat during the spec week and record the result for future RFC reference.
4. **Hyphae/RFC-0001 production-readiness (Critic gap).** Not blocking this RFC (Phase 2 deferred), but the lead must verify RFC-0001's actual status (design / prototype / shipped) and append the answer to this RFC's "Prior art" section before this RFC is marked `accepted`. If RFC-0001 is design-only, Phase 2 "live diagram" is also at-risk and we should not implicitly promise it.
5. **Audience definition (Critic gap).** This RFC commits to: **primary** audience = human dev on README/HN looking at the screenshot; **secondary** audience = AI agent calling `viz action=demo` from another tool. The demo intentionally optimizes for the primary at the cost of the secondary (a 600KB HTML response is fine for a human, expensive for an agent — that is OK; agents should use `nav` and other facades for routine graph queries, not `demo`).
6. **Frontend library choice (Critic gap: "1-week prototype-spike just to choose").** Candidates: Cytoscape.js (mature, ~600KB), vis-network (~400KB, simpler theming), react-flow (lock-in to React build pipeline → No), D3 force layout (most flexible, most code to write). Lead spikes the simplest possible 50-edge demo on Cytoscape during Phase 0 (1 day, time-boxed). If it does not produce a readable, tooltip-supporting page in one day, fall back to vis-network. **No** SPA build pipeline; deps inlined as `<script>` tags.
7. **What was wrong with the LAST viz round (RFC-0015)? (Critic gap, the most uncomfortable one.)** RFC-0015 shipped 6 UML diagram subtypes in v1.23.0; ROADMAP-beyond-codegraph.md still does not list viz as P0-P4. Why didn't shipping 6 diagram types move adoption/stars? Honest answer required before this RFC ships. Lead's working hypothesis: RFC-0015 shipped *capability* (6 diagram emitters) without a *hero* (no marketing artifact, no README screenshot, no shareable URL). RFC-0020 deliberately inverts this: it ships *one hero* and treats the capability as a means to that hero. If this hypothesis is wrong, RFC-0020 will fail for the same reason RFC-0015 did, and we will at least learn it cheaply.
